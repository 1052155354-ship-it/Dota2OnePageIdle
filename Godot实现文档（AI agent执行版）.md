# DOTA2主题一页挂机放置小游戏——Godot实现文档（AI agent执行版）

本文件不是策划文档，也不是程序岗位规范文档。

本文件的用途只有一个：
**让 AI agent 或程序在 Godot 中直接把这个原型搭出来。**

写法尽量使用自然语言，但会把必须写死的规则写死，避免实现时自由发挥过头。

本文件以仓库中的《策划文档.md》和《精确数值表（策划文档基准版）.md》为准。若三者冲突，以最新的数值表和本实现文档为准。

---

## 一、先把项目目标钉死

这个项目不是 Dota2 对局模拟器，也不是塔防游戏，更不是需要地图移动和战斗动画的 ARPG。

这个项目要做的是一个：

- 单场景
- 单页面
- 自动运行
- 少量手动决策
- 文本日志持续播报
- 通过成长最后推爆基地的挂机放置小游戏

AI agent 在实现时必须遵守以下边界：

- 不做地图
- 不做英雄移动
- 不做敌方英雄实体演出
- 不做复杂技能施法
- 不做失败条件
- 不做复杂结算系统
- 不做多页面来回切换
- 不要把项目扩大成真正的 Dota-like

整个游戏从开始到胜利，都应该尽量停留在同一个主场景里完成。

## 二、最终要做出来的画面是什么

整个游戏只有一个主页面。

页面上同时承载以下信息：

### 1. 顶部区域

用于显示整局比赛的全局信息。

要显示：

- 当前比赛时间
- 敌方基地总血条
- 敌方基地剩余百分比

顶部区域的任务是让玩家一眼看见：

- 现在打了多久
- 离胜利还有多远

### 2. 中央区域

用于显示炼金术士主体。

这里可以先放：

- 炼金术士立绘
- 或者一个占位图
- 或者一个简单角色框

这个区域的重点不是动画，而是作为页面视觉中心。

### 3. 左侧信息区

用于显示当前角色状态。

要显示：

- 当前等级
- 当前经验值
- 当前升级进度
- 当前金币
- 当前策略
- 累计补刀数
- 累计击杀数
- 大招当前状态（冷却中 / 生效中 / 未解锁）

### 4. 右侧装备区

用于显示 6 个固定装备槽位。

固定显示的装备：

- 点金手
- 假腿
- 狂战斧
- 隐刀
- 玲珑心
- 强袭

显示规则：

- 已购买装备：常亮
- 未购买且当前买不起：常暗
- 未购买但当前买得起：高亮，并在自动暂停时允许点击选择

这里不要做动态装备列表，不要生成“背包系统”，就是 6 个固定槽位。

### 5. 下方技能区

用于显示 4 个技能位置。

固定显示：

- 金钱强化
- 补刀强化
- Gank强化
- 爆发强化（大招）

显示规则：

- 已选技能：常亮
- 未选技能：常暗
- 当前可选技能：高亮

### 6. 底部日志区

这是表现层的关键部分。

页面下方需要有一个可滚动文本区域，不断输出事件日志。

日志至少要能显示：

- 补刀
- Gank
- 升级
- 技能选择
- 装备购买
- 大招开启
- 基地血量下降
- 胜利

本项目没有战斗动画，所以日志必须承担“比赛正在发生”的表现责任。

### 7. 暂停选择弹窗

平时不显示。

在两种情况下弹出：

- 升级时弹出技能选择
- 金币达到装备门槛时弹出装备选择

弹窗出现时，游戏运行暂停；选择完成后，游戏继续。

## 三、Godot 场景怎么搭

建议只做一个主场景，名称可以直接用：

- `main.tscn`

这个场景作为整个原型的唯一核心场景。

主场景里建议至少有以下几个大块节点：

- 一个总根节点，用来挂总控脚本
- 一个 UI 根节点，用来承载所有页面显示元素
- 一个计时与系统节点组，用来放运行逻辑所需的 Timer
- 一个弹窗层，用来显示暂停选择面板

如果用自然一点的话，可以理解为：

- 最外层负责整局控制
- 中间一层负责显示页面
- 再下面一层负责逻辑计时
- 最上面覆盖一层弹窗

不要把这个项目拆成很多场景页面反复切换。

本项目最适合的实现方式就是：
**一个主场景 + 几个可复用 UI 容器 + 若干脚本。**

## 四、程序里必须保存哪些状态

程序运行时，需要一直维护一份当前对局状态。可以做成一个总状态对象，也可以集中放在总控脚本里，但内容必须完整。

至少要保存以下字段：

### 基础运行状态

- 当前比赛时间
- 游戏是否暂停
- 当前是否已经胜利
- 当前弹窗类型（无 / 技能选择 / 装备选择）

### 角色成长状态

- 当前等级
- 当前经验
- 当前金币
- 当前补刀数
- 当前击杀数

### 战局状态

- 敌方基地当前生命值
- 敌方基地最大生命值

### 策略状态

- 当前策略（补刀 / Gank / 推塔）
- 策略切换冷却剩余时间

### 技能状态

- 金钱强化是否已获得
- 补刀强化是否已获得
- Gank强化是否已获得
- 大招是否已解锁

### 大招运行状态

- 大招是否正在生效
- 大招冷却剩余时间
- 大招持续剩余时间

### 装备状态

- 点金手是否已购买
- 假腿是否已购买
- 狂战斧是否已购买
- 隐刀是否已购买
- 玲珑心是否已购买
- 强袭是否已购买

程序实现上不要省掉这些字段，不要只靠 UI 文本反推状态。

## 五、开局流程必须这样跑

主场景载入后，按下面顺序执行：

### 第一步：初始化整局状态

把所有初始值设置成数值表规定的值：

- 等级 = 1
- 经验 = 0
- 金币 = 0
- 补刀数 = 0
- 击杀数 = 0
- 当前策略 = 补刀
- 基地生命 = 1000
- 比赛时间 = 0
- 胜利 = false
- 游戏暂停 = true

### 第二步：刷新一次全部 UI

此时页面虽然还没开始跑，但所有文本、血条、装备槽、技能槽都应先显示正确初始状态。

### 第三步：立刻弹出第一次技能选择

因为策划文档规定：

- 初始等级就是 1 级
- 进入游戏后，1级即可进行第一次技能选择

所以程序不应该等经验增长后再给第一次技能。

开局就暂停，并弹出技能选择弹窗。

此时可选技能应为三个小技能：

- 金钱强化
- 补刀强化
- Gank强化

玩家选完其中一个后：

- 记录技能已获得
- 关闭弹窗
- 写一条日志
- 游戏进入正式运行状态

## 六、正常运行时，整局怎么循环

正式开始后，游戏会在运行状态下持续循环。

### 1. 比赛时间持续增加

只要游戏没有暂停、没有胜利，比赛时间就持续向前走。

### 2. 当前策略持续结算

当前策略只能是三种之一：

- 补刀
- Gank
- 推塔

程序不需要让三种行为同时发生，只结算当前选中的那一种。

### 3. 每次行为结算后都要做四件事

- 更新资源或基地血量
- 写日志
- 刷新 UI
- 检查是否触发升级 / 装备 / 胜利

### 4. 如果触发暂停事件，就中断正常循环

触发暂停的事件只有两种：

- 升级选择
- 装备选择

暂停时所有行为循环、比赛时间推进、大招冷却、策略切换冷却，都应该一起停住。

### 5. 胜利后停止一切正常循环

基地生命值降到 0 时：

- 立即进入胜利状态
- 停止行为结算
- 停止计时
- 写胜利日志
- 弹出一个简易胜利提示或结算面板

## 七、三种策略具体怎么实现

### 1. 补刀策略

当当前策略为补刀时：

程序按“实际补刀间隔”不断触发补刀结算。

每次补刀结算时：

- 金币增加
- 经验增加
- 补刀数加一
- 写一条普通日志

补刀的实际间隔和单次金币，要按数值表结合技能和装备实时计算。

补刀相关加成来源包括：

- 金钱强化
- 补刀强化
- 点金手
- 假腿
- 狂战斧
- 大招补刀加速

这里不要把数值写死在脚本流程里，应该统一从当前状态计算出结果。

### 2. Gank 策略

当当前策略为 Gank 时：

程序按“实际 Gank 周期”不断触发 Gank 结算。

每次 Gank 结算时：

- 金币增加
- 经验增加
- 击杀数加一
- 写一条比补刀更重要一点的日志

Gank 相关加成来源包括：

- Gank强化
- 假腿
- 隐刀
- 大招 Gank 加速

### 3. 推塔策略

当当前策略为推塔时：

程序持续对敌方基地造成伤害。

实现上最简单的方式是：

- 用固定频率，例如每 0.2 秒或每 0.1 秒结算一次伤害
- 或者每帧按 delta 扣血

但无论用哪种方式，最终伤害要等价于数值表中的“每秒推塔伤害”。

推塔时：

- 不加金币
- 不加经验
- 不增加补刀数
- 不增加击杀数
- 只减少基地生命值
- 周期性写日志，不要每一小步都刷一条日志

推塔相关加成来源包括：

- 假腿
- 强袭
- 大招推塔倍率

开局直接推塔应该几乎没效率，这一点一定要通过数值计算自然体现出来，而不是手写硬限制。

## 八、策略切换怎么处理

页面上要有三个策略按钮：

- 补刀
- Gank
- 推塔

玩家点击其他策略按钮时，按下面规则处理：

- 如果当前正在暂停，不允许切换
- 如果策略切换冷却未结束，不允许切换
- 如果点击的是当前已在使用的策略，不做任何处理
- 其他情况立即切换到新策略
- 切换后把策略切换冷却设置为 5 秒
- 写一条策略切换日志
- 刷新当前策略显示和按钮高亮

策略切换冷却在正常运行时递减，暂停时停止递减。

为了减少实现复杂度，策略切换后可以直接重置当前策略对应的行为计时，不必继承上一个策略剩余计时进度。这样更直观，也更稳定。

## 九、升级系统怎么做

升级是本项目最重要的暂停节点之一。

### 1. 升级条件

当当前经验达到升级阈值时，立刻触发升级暂停。

阈值按数值表执行：

- 到 2 级累计 120
- 到 3 级累计 340
- 到 4 级累计 720

### 2. 升级触发时要做什么

- 游戏暂停
- 打开技能选择弹窗
- 高亮当前可选技能
- 等待玩家点击

### 3. 当前可选技能的规则

- 1级开局时，可从三个小技能里选 1 个
- 升到 2 级时，可从剩下的两个小技能里选 1 个
- 升到 3 级时，自动只剩最后一个小技能可选，也仍然弹出一次选择流程
- 升到 4 级时，只可选大招爆发强化

不要把 3 级时最后一个小技能直接无提示自动给掉，仍然要保留“暂停、显示、点击确认”的流程，这样玩家感受更统一。

### 4. 选择完成后

- 标记技能已获得
- 若是 4 级，则同时标记大招已解锁并准备开始自动循环
- 写一条关键日志
- 刷新技能区 UI
- 关闭弹窗
- 恢复游戏运行

## 十、装备系统怎么做

装备系统同样是自动暂停选择。

### 1. 装备固定，不是掉落，不是商店页

页面右侧始终显示 6 个固定装备槽。

程序里也固定管理这 6 件装备，不要做随机装备池。

### 2. 装备可选条件

只要满足：

- 该装备还未购买
- 当前金币大于等于该装备价格

它就属于“当前可购买装备”。

### 3. 自动暂停规则

当游戏处于正常运行状态，并且存在至少一件可购买装备时：

- 若当前没有升级事件优先级压住它
- 则自动暂停
- 打开装备选择弹窗
- 高亮所有当前买得起且未购买的装备

### 4. 装备选择时怎么处理

玩家点击其中一件后：

- 扣除对应金币
- 标记装备已购买
- 立即应用装备效果
- 写一条关键日志
- 刷新装备区 UI
- 关闭弹窗

### 5. 购买后再次检查

如果购买完这一件后，剩余金币仍然还能买其他未购买装备：

- 游戏恢复后稍微延迟一下再次检查
- 然后重新暂停并弹出装备选择

这样可以符合“达到装备门槛就自动暂停选择”的策划要求。

### 6. 装备顺序不要写死

虽然文档里有推荐顺序，但程序不应强制顺序。

AI agent 在实现时，必须允许玩家自由决定购买顺序，只要金币够即可。

## 十一、大招系统怎么做

大招是 4 级后自动运行的一个周期系统。

### 1. 解锁前

- 大招未激活
- 不进入冷却循环
- 技能区显示未解锁

### 2. 升到 4 级并选择大招后

开始进入大招自动循环。

程序最简单的做法是维护两个状态：

- 大招是否正在生效
- 大招剩余持续时间 / 冷却时间

### 3. 运行规则

- 若大招未生效且冷却完成，则自动开启
- 开启后进入持续阶段
- 持续结束后进入冷却阶段
- 冷却结束后再次自动开启

### 4. 玲珑心的影响

如果已购买玲珑心：

- 大招持续时间从 12 秒变成 16 秒
- 大招冷却时间从 45 秒变成 30 秒

因此大招系统的持续和冷却，不应该写成固定常量死锁在初始化里，而应根据当前装备状态动态取得。

### 5. 大招开启时的表现

- 写一条明显的里程碑日志
- 左侧状态区显示“大招生效中”
- 页面可以做一点简单高亮，比如角色框发光、技能图标高亮、顶部出现文字提示

不需要复杂特效，有存在感即可。

---

## 十二、日志系统怎么做

这个项目的日志系统很重要，不要把它当成附属功能。

### 1. 日志必须支持持续追加

新日志出现时：

- 追加到日志列表底部
- 日志区自动滚到最新内容

### 2. 日志分三层

#### 普通事件

例如：

- 补到一个兵，获得 X 金币
- 正在推进基地

#### 关键事件

例如：

- 完成一次 Gank
- 升到 2 级 / 3 级 / 4 级
- 购买一件装备
- 切换策略

#### 里程碑事件

例如：

- 解锁大招
- 大招开启
- 基地血量跌破 75% / 50% / 25% / 10%
- 最终胜利

### 3. 日志文本风格

可以写得像比赛播报和挂机结算混合体。

例如：

- 炼金术士补到一个小兵，金币 +3，经验 +1
- 炼金术士完成一次 Gank，金币 +26，经验 +34，击杀数 +1
- 炼金术士升到 2 级，正在选择新技能
- 炼金术士购买了点金手，刷钱效率提升
- 敌方基地血量跌至 50%，比赛进入后期
- 炼金术士火力全开，爆发强化启动
- 敌方基地被摧毁，比赛胜利

不需要一开始就写很多文案分支，先把事件覆盖完整最重要。

## 十三、UI 刷新规则一定要稳定

本项目几乎所有反馈都来自 UI，所以刷新必须明确。

### 每次以下情况发生后，立刻刷新对应 UI

#### 金币变化后

刷新：

- 金币文本
- 装备区高亮状态

#### 经验变化后

刷新：

- 等级文本
- 经验文本
- 经验条

#### 补刀或击杀变化后

刷新：

- 补刀数文本
- 击杀数文本

#### 策略变化后

刷新：

- 当前策略文本
- 三个策略按钮的高亮状态
- 策略切换冷却提示

#### 技能获得后

刷新：

- 技能区图标状态
- 可选技能高亮状态

#### 装备购买后

刷新：

- 装备区图标状态
- 可选装备高亮状态

#### 大招状态变化后

刷新：

- 大招状态文本
- 技能区大招图标状态
- 可能的角色区域高亮表现

#### 基地血量变化后

刷新：

- 基地血条
- 百分比文本

不要把 UI 刷新完全塞进 `_process` 每帧全量刷新。这个项目更适合：

- 状态变化时主动刷新
- 少数持续变化内容，比如时间、血条插值，可按需要局部更新

## 十四、暂停系统必须怎么理解

这个项目的暂停不是菜单暂停，而是“玩法流程暂停”。

### 暂停时要停掉的东西

- 比赛时间推进
- 当前策略行为结算
- 策略切换冷却
- 大招持续与冷却计时
- 任何自动检查流程

### 暂停时不需要停掉的东西

- UI 显示
- 弹窗交互
- 鼠标点击
- 日志阅读

### 恢复时怎么处理

玩家完成一次技能或装备选择后：

- 关闭弹窗
- 清除当前暂停原因
- 恢复游戏运行
- 再开始正常循环

### 优先级规则必须写死

若同一时刻既满足升级又满足装备可购买：

- 先进入升级选择
- 升级选完并恢复后
- 再重新检查装备是否需要暂停

这个优先级不要交给 agent 自己理解，必须按此实现。

## 十五、胜利流程怎么做

当敌方基地生命值小于等于 0 时：

- 把基地生命值钳制到 0
- 标记胜利 = true
- 标记游戏暂停 = true
- 停止所有运行计时
- 写最终胜利日志
- 显示一个简单的胜利面板

胜利面板不需要复杂评分。

先显示这些信息就够：

- 胜利文字
- 总比赛时间
- 最终等级
- 最终金币
- 补刀数
- 击杀数
- 已购买装备数

再给一个按钮：

- 重新开始

重新开始后，直接重置本局状态并回到开局第一次技能选择。

## 十六、脚本应该怎么拆

这个项目不需要拆得特别碎，但也不要所有逻辑堆进一个脚本。

推荐拆成下面几类：

### 1. 总控脚本

负责：

- 初始化
- 暂停与恢复
- 胜利判断
- 调用各系统
- 统一刷新 UI

这会是主场景最重要的脚本。

### 2. 运行状态脚本或数据对象

负责保存：

- 当前等级
- 当前经验
- 当前金币
- 当前策略
- 当前装备状态
- 当前技能状态
- 当前基地生命
- 当前比赛时间

### 3. 行为结算脚本

负责：

- 补刀计算
- Gank 计算
- 推塔计算
- 策略切换

### 4. 技能与升级脚本

负责：

- 升级阈值检查
- 技能可选列表
- 技能选择后应用效果

### 5. 装备脚本

负责：

- 装备价格判断
- 装备可选列表
- 装备购买后应用效果

### 6. 大招脚本

负责：

- 大招自动循环
- 冷却与持续
- 玲珑心修正

### 7. 日志脚本

负责：

- 追加日志
- 自动滚动到底部

### 8. UI 脚本

负责：

- 将运行状态同步到页面显示
- 控制高亮、常亮、常暗

AI agent 实现时可以按这个职责拆脚本，但不用拘泥于我写的文件名，只要职责边界清楚即可。

## 十七、最推荐的开发顺序

请按下面顺序实现，不要一上来追求全做完。

### 第一步：先把空页面搭出来

先把这些 UI 放出来：

- 顶部时间和基地血条
- 左侧状态区
- 中央角色区
- 右侧 6 个装备槽
- 下方 4 个技能槽
- 底部日志区
- 暂停弹窗

此时哪怕全是占位内容也没关系。

### 第二步：实现基础状态和 UI 刷新

让以下内容能正确显示：

- 时间
- 金币
- 经验
- 等级
- 补刀
- 击杀
- 基地血量
- 当前策略

### 第三步：实现补刀策略

先只做补刀，让游戏可以：

- 自动补刀
- 获得金币和经验
- 刷日志
- 刷 UI

### 第四步：实现升级暂停与技能选择

确保：

- 开局先弹第一次技能选择
- 升级时会暂停并选技能
- 选完后继续运行

### 第五步：实现装备自动暂停购买

确保：

- 金币到门槛时自动暂停
- 装备可选项高亮
- 买完后立即生效

### 第六步：实现 Gank 和推塔策略

加上：

- 策略按钮切换
- Gank 循环
- 推塔扣基地血

### 第七步：实现大招自动循环

确保：

- 4级后自动生效
- 持续 / 冷却正确循环
- 玲珑心能改变大招参数

### 第八步：实现胜利流程

基地归零时正确结束整局。

### 第九步：补表现细节

最后再补：

- 高亮状态
- 颜色区分
- 简单特效
- 更顺眼的日志排版

## 十八、对 AI agent 的最后限制

AI agent 在实现时，必须优先保证：

- 逻辑闭环正确
- 数值计算正确
- 暂停流程正确
- UI 状态正确

不要优先追求：

- 复杂动画
- 复杂架构
- 花哨特效
- 额外系统扩展

如果某处不确定，优先遵守下面这条：

**宁可做得简单直白，也不要额外发明新系统。**

这个项目的正确方向是：

- 页面简洁
- 逻辑清楚
- 运行稳定
- 能完整从开局跑到胜利

只要能做到这一点，这个原型就是成功的。
